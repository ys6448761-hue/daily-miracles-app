---

## 1️⃣ 존재 이유

DreamTown은
사람을 소비시키는 플랫폼이 아니라,
사람이 자기 삶을 다시 살아가게 하는 세계다.

---

## 2️⃣ 핵심 5축

🌱 소원꿈터
= 마음과 별이 자라는 개인 우주

🌿 일상여정
= 삶의 숨결 회복

🌊 소원여정
= 삶의 전환점과 깊은 항해

🌌 나의 별자리
= 시간 속 마음의 기록

💌 공명
= 조용한 연결과 집단지성

---

## 3️⃣ 금지 원칙

DreamTown은 다음 구조를 절대 만들지 않는다.

❌ 중독 구조
❌ 비교 구조
❌ 경쟁 구조
❌ 감정 소비
❌ 성장 강박

이 5가지 중 하나라도 강화되는 기능은
DreamTown 기능이 아니다.

---

## 4️⃣ 중심 원칙

DreamTown의 모든 기능은 다음을 중심으로 한다.

⭕ 의미 발견
⭕ 삶의 감각 회복
⭕ 조용한 연결
⭕ 공명
⭕ 시간의 축적

---

## 5️⃣ DreamTown UX 기준 (회귀 테스트 5항목)

모든 정체성 P0 작업은 다음 5항목을 통과해야 한다.

1. 감정이 먼저 오는가?
2. 별이 결과로 느껴지는가?
3. 아우룸이 전달자인가?
4. 관광보다 변환이 느껴지는가?
5. 공유 전에 감정 여운이 남는가?

→ 단순 작동 검증 X
→ DreamTown답나? 통과 강제

---

## 6️⃣ P0 분류 (3종)

🔴 정체성 P0 (최우선)
- DreamTown이 DreamTown 아니게 됨
- 예: 감정 흐름 삭제, 아우룸 붕괴, 세계관 충돌
- 무게: 다른 P0보다 항상 우선

🟠 시스템 P0
- 서비스 작동 문제
- 예: migration, API, SSR
- 무게: 기능 유지 핵심

🟡 운영 P0
- 효율/속도/자산
- 예: anchor push, 자산 정리, GPTs
- 무게: 후순위 가능

→ 모든 P0 발견 시 분류 강제
→ 발견 ≠ 즉시 처리 (P0 인플레이션 회피)

---

## 7️⃣ 변경 원칙

이 헌법의 변경은 CEO 승인이 반드시 필요하다.

- 추가: CEO 승인 후 v1.1, v1.2...
- 삭제: CEO 승인 후 DEC 문서 작성
- 수정: 단어 하나도 CEO 검토 필수

→ 헌법 = 흔들리지 않는 기준
→ 임의 변경 절대 금지

## 부속: DreamTown 정체성 5기준 (참조)

기존 메모리 SSOT와 정합:

① 보이지 않는 마음이 출발점인가
② 이미지 형상화가 마음 시각화인가 (풍경 X)
③ 별이 결과물인가
④ 함께 성장 메커니즘이 있는가
⑤ 동반 관계인가

→ 5기준 ⊃ 5축 ⊃ UX 5항목
→ 상위 → 하위 일관성

## 🌟 최종 판단 기준 (헌법 영혼)

DreamTown의 모든 기능과 결정은
"이게 사람의 삶을 조금 더 살아있게 만드는가?"
를 최종 기준으로 판단한다.

이 한 질문이
헌법 7개 항목과
모든 부속의 위에 있다.

의심스러울 때:
→ 다른 기준을 보기 전에
→ 이 질문 먼저
→ 답이 "아니오"면 어떤 기능도 만들지 않는다
