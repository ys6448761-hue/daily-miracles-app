## 3. 리스크 시나리오 및 대응

### 시나리오 1: 외부인이 데이터 변경

**상황**: 공유받은 사람이 테스트 데이터를 수정/삭제

**영향**: 다른 데모에 영향

**대응**:
1. Reset API로 데이터 복구 (내부용)
2. 중요 데모 전 항상 리셋 실행
3. 데모 후에도 리셋 권장

**예방**:
- "읽기만 해주세요" 안내
- 중요 데모는 별도 행사 ID 사용 (향후)

### 시나리오 2: Reset Token 유출

**상황**: 내부 문서가 외부로 유출되어 토큰 노출

**영향**: 데이터 무단 삭제 가능

**대응**:
1. 즉시 토큰 변경 (ENV 수정)
2. 서버 재배포
3. 유출 경로 조사

**예방**:
- 토큰 포함 문서는 "INTERNAL ONLY" 표기
- 외부 공유 시 토큰 섹션 삭제

### 시나리오 3: 경쟁사가 시스템 분석

**상황**: 경쟁사가 데모를 통해 기능 분석

**영향**: 기능 복제 가능성

**대응**:
- 법적 대응 어려움 (공개 데모)
- 차별화 포인트 지속 개발

**예방**:
- 핵심 로직은 백엔드에서 처리
- 데모는 결과만 보여주고 과정은 설명

### 시나리오 4: 서버 장애 중 데모 요청

**상황**: 외부 미팅 중 서버 다운

**영향**: 신뢰도 하락

**대응**:
1. 사전 녹화 영상 재생
2. 스크린샷으로 설명
3. "서버 점검 중, 추후 안내" 멘트

**예방**:
- 중요 미팅 전 헬스체크
- 백업 자료 항상 준비

## 4. 민감 정보 체크리스트

### 공유 전 확인

- [ ] Reset Token 포함 여부 → 제거
- [ ] Event UUID 노출 여부 → 가능하면 제거
- [ ] API URL 직접 노출 → 메인 URL로 대체
- [ ] 내부 연락처 → 공식 이메일로 대체
- [ ] 가격/비용 정보 → 결재 후에만 공유

- [ ] 브라우저 탭에 민감 URL 없음

- [ ] 콘솔 로그 없음
- [ ] 테스트 데이터 사용 (실제 이름 X)
