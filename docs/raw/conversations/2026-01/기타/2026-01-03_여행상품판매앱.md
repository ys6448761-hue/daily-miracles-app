## 📌 프로젝트 개요

### 목표
Wix 홈페이지에 연동할 **여수 소원항해 1박2일 투어패스 견적 시스템** 개발

### 문제점 (현재)
- 옵션 조합이 너무 많음 (인원 × 요일 × 호텔 × 레저 = 320가지+)
- Wix에서는 복잡한 옵션 상품 구성이 어려움
- 고객이 가격 확인하려면 일일이 문의해야 함

### 해결책
- 외부 견적 시스템 웹앱 개발
- 옵션 선택 → 실시간 가격 계산 → 예약 요청
- Wix에서 iframe 또는 링크로 연동

CREATE TABLE quote_requests (
  id SERIAL PRIMARY KEY,

  leisure_price INTEGER,
  voyage_price INTEGER,
  total_price INTEGER,
  
  -- 고객 정보
  customer_name VARCHAR(100),
  customer_phone VARCHAR(20),
  customer_email VARCHAR(100),
  travel_date DATE,
  memo TEXT,
  
  -- 상태
  status VARCHAR(20) DEFAULT 'pending',      -- pending, confirmed, cancelled
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()

## 🔄 예약번호 생성 규칙

javascript
function generateQuoteNumber() {
  const prefix = 'SW';  // SoWon (소원)
  const date = new Date().toISOString().slice(0,10).replace(/-/g, '');
  const seq = getNextSequence();  // 001, 002, 003...
  
  return `${prefix}-${date}-${seq.toString().padStart(3, '0')}`;
  // 예: SW-20250103-001
}

2) “원가/마진/판매가/정상가” 표기: 어떻게 설계하는 게 좋나?
핵심은 두 레이어로 분리입니다.

A. 내부(운영/정산) 레이어: 원가/마진/판매가가 필요
각 항목(숙박/레저/체험/운영비)에 아래 필드를 두면 계산이 깔끔해져요.

✅ 내부 데이터 구조(권장)
cost: 원가(실제 지출/정산 기준)

sell: 판매가(고객 결제에 반영)

msrp/list: 정상가(“혜택 가치” 산출용, 선택)

margin = sell - cost

margin_rate = margin / sell

여행사 차지는 아주 단순:

cost=0, sell=20,000(1인), msrp는 필요 없음

이렇게 하면

“총마진” = 각 항목 margin 합

“할인”을 넣어도 손익 방어가 쉬움(어디서 깎았는지 추적 가능)

B. 고객(표시) 레이어: “원가/마진”은 보여주지 말기
고객에게 원가/마진을 직접 보여주면 신뢰보다 가격 민감/흥정/클레임이 커지는 경우가 많아요.

대신 고객 화면에는 아래 정도만 추천:

✅ 고객 표시 추천 조합
구성 금액(결제 반영)

숙박 / 레저 / 체험 / 여행 운영비 / 총액

혜택(추가 제공)

무료 혜택 가치(정상가 기반)

“차감 없음” 문구 고정

(선택) 정상가 대비 혜택

“정상가 기준 혜택 21,000원 상당 제공” 같은 형태
(단, 정상가 산정 기준을 아주 짧게라도 명시하면 더 안전)

3) 1박2일 상품에서 “정상가”는 뭘로 잡는 게 좋나?
정상가(list/msrp)는 “혜택 가치”를 뒷받침하는 기준이라서 논리적으로 일관해야 해요.

추천 우선순위:

공식 판매처/현장가/정가(티켓 정가 등)

제휴처 제공 “정상가 표”가 있으면 그걸 기준

없으면 내부적으로는 표준 정가 테이블을 별도로 둬서 시즌마다 업데이트

핵심: 정상가는 “할인처럼 보이게”가 아니라 혜택/가치 커뮤니케이션용으로만 쓰는 게 안전합니다.
