# 이야기가 책이 되는 설계 구조
### + 루미 기획과의 충돌 여부 분석

*2026-04-04 · 루미를 위한 설명 자료*

---

## 1부. 어떻게 설계되어 있는가

### 핵심 원리: "쌓이면 된다"

소원이가 하는 일은 단 하나다.
**매일 별(소원)과 대화를 기록한다.**

그 기록들이 정해진 구조 안에 쌓이면,
나중에 꺼내서 책으로 만들 수 있다.

---

### 실제 저장 구조 (DB 테이블)

```
소원 등록
  └─ dt_wishes
       └─ wish_text ("나는 내 삶을 스스로 결정하는 사람이 되고 싶다")
       └─ yeosu_theme ("여수 소원 의식 참여" 같은 특별 진입 경로)

       ↓ 소원을 기반으로 별 탄생

  dt_stars (별 = 소원의 여정 전체)
       └─ star_name, star_stage, galaxy (성장/도전/치유/관계 중 하나)
       └─ created_at → 이날이 D+1

       ↓ 매일 Aurora5가 케어하며 기록

  dt_dream_logs (모든 이벤트의 원장 — append-only)
       └─ log_type: origin / wisdom / choice / voyage / growth / artifact
       └─ payload: { message_text, day, emotion, ... }

  aurora5_messages (케어 메시지)
       └─ message, wisdom_tag, created_at

  dt_voyage_logs (사용자 감정/선택 기록)
       └─ emotion, tag, growth, wisdom_tag

  dt_choice_logs (중요 선택의 순간)
       └─ choice_type, choice_value

  dt_growth_reports (성장 보고서 — 주/월/분기/년)
       └─ period, summary, pattern, change_point

  dt_narrative_logs (이야기 서사 재구성)
       └─ chapter, title, content

  dt_artifact_jobs (소원그림 — AI 생성 이미지)
       └─ type: image / pdf / webtoon / video
       └─ result_url → 책의 삽화가 됨
```

---

### 타임라인 API가 하는 일

`GET /api/dt/engine/star/:id/timeline`

이 API 하나가 **7개 스트림을 동시에 꺼내서** 조합한다:

| 스트림 | 데이터 | 책에서의 역할 |
|--------|--------|--------------|
| star | 별 이름, 소원 문장, 은하 | 표지 + 제목 |
| dream_log | 전체 이벤트 200개 | 전체 서사 |
| wisdoms | 핵심 통찰 10개 | 인용구 |
| choices | 중요 선택 20개 | 전환점 |
| reports | 성장 보고서 5개 | 챕터 요약 |
| chapters | 서사 재구성 | 챕터 본문 |
| artifacts | 소원그림들 | 삽화 |

→ 이 7개가 합쳐지면 **편집만 하면 되는 원고**가 된다.

## 2부. 루미 기획과의 충돌 분석

### 루미가 기획하는 것 (파악된 내용 기준)

| 루미 기획 항목 | 내용 |
|--------------|------|
| A. 여수 소원 의식 체험 | 현지 소원 입력 → 별 탄생 특별 진입 경로 |
| B. 단체 별 탄생 축하 이벤트 | 매주 수요일, 그 주에 탄생한 별들 모아서 축하 |
| C. 여행 패키지 연계 | 소원 의식 + 단체 여행 + 다음날 결제 오픈 |

---

### 충돌 분석 결과

#### ✅ A. 여수 소원 의식 — **충돌 없음**

여수에서 소원을 입력하면 `yeosu_theme` 값이 소원(wish)에 저장된다.
별 생성 시에는 gem_type → galaxy 매핑만 사용하므로,
yeosu_theme은 소원 단계에서만 활용되고 별의 구조에는 영향을 주지 않는다.

```
현지 소원 입력
  → dt_wishes.yeosu_theme = "yeosu_ceremony"
  → dt_stars 생성 (일반 별과 완전히 동일한 구조)
  → D+1부터 동일한 Aurora5 케어 시작
  → 이야기가 쌓임 → 나중에 책이 됨
```

**오히려 시너지**: 여수에서 시작한 별이 시간이 지나 책이 되면,
"여수에서 시작한 소원이 책이 됐다"는 서사가 생긴다.

---

#### ✅ B. 단체 별 탄생 축하 이벤트 — **충돌 없음**

단체 이벤트는 별 탄생 시점(created_at)을 기준으로 그룹화한다.
별의 내부 여정(dt_dream_logs, 타임라인)은 개인별로 완전히 분리되어 있다.

```
수요일 단체 이벤트:
  → 그 주 탄생한 별들 조회 (SELECT WHERE created_at 기준)
  → 축하 메시지 발송
  → 각자의 별은 독립적으로 여정 시작

개인 여정은 각자의 D+1부터 독립 타임라인
```

**충돌 없음.** 단체 이벤트는 "시작을 축하하는 것"이고,
책 구조는 "시작 이후를 기록하는 것"이라 타이밍이 겹치지 않는다.

---

#### ✅ C. 여행 패키지 + 다음날 결제 오픈 — **충돌 없음**

여행 결제(9,900원 / 24,900원 / 89,000원)는
별 여정의 시작을 활성화하는 단계다.

책 전환(D+180 이후, 300~500만원)은
별 여정이 충분히 쌓인 후에 열리는 단계다.

```
여행 패키지 결제 → 별 활성화 → 여정 시작
                                  ↓
                            (6개월 이후)
                                  ↓
                          책 전환 CTA 오픈
```

**타이밍이 완전히 다르므로 충돌 없음.**

---

#### ⚠️ 주의할 것 하나

현재 book/upgrade, book/inquiry 신청이
**dt_dream_logs 테이블의 'voyage' 타입**으로 저장된다.

여수 이벤트 참여 기록도 같은 테이블에 저장될 경우,
나중에 분석할 때 구분이 필요하다.

→ JSONB payload의 `event` 필드로 구분하면 되고 (이미 설계됨),
→ 당장 문제가 되지는 않지만,
→ 루미 기획 중 이벤트 참여를 별도 집계해야 한다면 필드 추가를 미리 논의하면 좋다.

## 3부. 루미에게 전달할 핵심 메시지

### "설계가 먼저 되어 있으니 기획을 그 위에 얹으면 된다"

드림타운의 설계는:
- 소원 → 별 → 이야기 축적 → 책

루미의 기획은:
- 여수 현지 진입 → 별 탄생 → 단체 이벤트 → 여행 패키지

이 두 흐름은 **서로 다른 레이어**에서 작동한다.

```
루미 기획: "어떻게 별을 탄생시킬 것인가" (입구)
설계 구조: "탄생한 별이 어떻게 책이 되는가" (출구)
```

입구와 출구가 다르다. 충돌하지 않는다.
루미가 좋은 입구를 만들수록,
더 좋은 이야기가 쌓이고,
더 좋은 책이 된다.

---

### 루미가 추가로 고려하면 좋은 것

| 항목 | 내용 |
|------|------|
| 여수 특별 별 라벨 | yeosu_theme 활용 → "여수에서 태어난 별" 뱃지 가능 |
| 수요일 이벤트 기록 | dt_dream_logs에 'origin' 타입으로 이벤트 참여 기록 가능 |
| 여행 후 첫 항해 연결 | 여행 귀가 후 D+1 케어 메시지 특별 버전 가능 |

---

*설계와 기획이 연결될 때 이야기가 더 풍부해진다.*
