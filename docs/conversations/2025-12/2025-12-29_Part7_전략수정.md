# Part 7: 전략 수정

## 배경

Part 6에서 "소원이별 독립 프로젝트"를 계획했으나,
최수민님 방식으로 자가 점검한 결과 전략 수정이 필요함을 발견

---

## 기존 전략 (Part 6)

```
소원이 등록
    ↓
독립 프로젝트 생성 (wishes/sowoni_XXX/)
    ↓
agent-creator로 소원이별 에이전트 생성
    ↓
project-creator로 폴더 구조 생성
```

**문제점:**
- 과도한 복잡성
- 유지보수 부담
- 컨텍스트 관리 원칙 위배

---

## 최수민님 방식 자가 점검

### 체크리스트

| 항목 | 결과 | 비고 |
|------|------|------|
| 컨텍스트 효율적? | X | 소원이별 폴더 과다 |
| 스크립트로 해결 가능? | O | 단순 데이터 저장 |
| 반복 패턴 있음? | O | 모든 소원이 동일 흐름 |
| AI 추론 필요? | X | 정형화된 작업 |

### 결론

> "소원이별 독립 프로젝트"는 과설계(over-engineering)
> 단순한 데이터 구조 + 배치 처리로 충분

---

## 수정된 전략

### Before (복잡)

```
wishes/
├── sowoni_001/
│   ├── profile.json
│   ├── analysis.json
│   ├── messages/
│   └── ...
├── sowoni_002/
└── ... (수백 개 폴더)
```

### After (단순)

```
data/
├── users.json           # 전체 소원이 목록
├── analyses/            # 분석 결과
│   └── 2025-12-29.json  # 날짜별 배치
├── messages/            # 생성된 메시지
│   └── 2025-12-29.json
└── logs/                # 발송 로그
    └── 2025-12-29.json
```

---

## 핵심 변경점

| 항목 | Before | After |
|------|--------|-------|
| 폴더 구조 | 소원이별 분리 | 날짜별 배치 |
| 에이전트 | 소원이별 생성 | 공용 에이전트 |
| 데이터 저장 | 개별 파일 | 배치 JSON |
| 복잡도 | 높음 | 낮음 |

---

## 새로운 흐름

```
소원이 등록
    ↓
data/users.json에 추가
    ↓
배치 분석 (10명씩)
    ↓
data/analyses/YYYY-MM-DD.json 저장
    ↓
배치 메시지 생성
    ↓
data/messages/YYYY-MM-DD.json 저장
    ↓
배치 발송
    ↓
data/logs/YYYY-MM-DD.json 로깅
```

---

## 불필요해진 것들

| 항목 | 상태 | 이유 |
|------|------|------|
| agent-creator | 보류 | 당장 불필요 |
| project-creator | 취소 | 단순 구조로 대체 |
| wishes/ 폴더 | 취소 | data/로 통합 |

---

## 교훈

### 최수민님 방식 적용 결과

1. **과설계 방지**
   - "화려한 구조"보다 "작동하는 단순함"
   - 복잡성은 필요할 때 추가

2. **자가 점검의 가치**
   - 실행 전 전략 검증
   - 잘못된 방향 조기 발견

3. **컨텍스트 관리 원칙 준수**
   - 배치 처리 (10개씩)
   - 파일 저장 (날짜별)
   - 스크립트 우선
