# 자가 검증 시스템

## 핵심 철학

> "내가 썼을 때 만족스러운가?" (감성적 기준)
> "에이전트 지침에 오류 방지 로직이 있는가?" (기술적 기준)

검증 없는 배포는 기술 부채를 쌓는 것이다.

---

## 4단계 검증 프로세스

### 1단계: 도그푸딩 (직접 사용)

> 만든 사람이 첫 번째 사용자가 된다

**체크포인트**:
- 내가 만든 서비스를 실제로 사용해봤는가?
- 사용하면서 불편한 점이 없었는가?
- 다른 대안 대신 이걸 계속 쓰고 싶은가?

**배드 시그널**:
- 내가 만든 서비스 대신 다른 걸 쓰고 싶다면? → 즉시 개선!
- 테스트용으로만 한번 써보고 안 쓴다면? → 근본 문제 있음

### 2단계: 스킬에 검증 로직 포함

> 스킬 만들 때 "검증도 할 수 있게" 설계

**예시**:
```markdown
## 출력 검증
작업 완료 후 다음을 확인하라:
1. 출력 형식이 스키마와 일치하는가?
2. 필수 필드가 모두 있는가?
3. 값의 범위가 적절한가?

문제 발견 시 → 재생성
```

### 3단계: 결과 기반 지침 보강

> 오류 발견 → 에이전트 .md에 즉시 반영

**프로세스**:
```
오류 발생 → 원인 분석 → 해결 → 지침에 영구 반영
                              ↓
                    "이 오류가 다시 발생하지 않도록"
```

**한 번 수정으로 끝내지 않는다. 지침에 영구 기록한다.**

### 4단계: 파이프라인 최종 종합

> 마지막에 "종합 검증" 단계

**파이프라인 예시**:
```
1. 데이터 수집
2. 분석
3. 콘텐츠 생성
4. ✅ 종합 검증 ← 여기서 전체 정합성 확인
5. 배포
```

## 팀원별 자가 검증 체크리스트

### 코미 (Chief Operating AI Manager)
- [ ] 보고서가 읽기 쉬운가?
- [ ] 지시가 명확하고 모호함이 없는가?
- [ ] 누락된 정보가 없는가?
- [ ] 수치와 날짜가 정확한가?
- [ ] 담당자가 명확히 지정되었는가?

### 재미 (Content AI)
- [ ] 콘텐츠에 감동/영감이 있는가?
- [ ] 브랜드 톤앤매너에 맞는가?
- [ ] 오탈자/문법 오류가 없는가?
- [ ] 대상 독자에게 적합한가?
- [ ] CTA가 명확한가?

### 루미 (Research AI)
- [ ] 분석 결과에 실행 가능한 인사이트가 있는가?
- [ ] 수치가 정확하고 검증되었는가?
- [ ] 출처가 명확한가?
- [ ] 비교 기준이 적절한가?
- [ ] 결론이 데이터에서 논리적으로 도출되었는가?

### 여의보주 (Inspiration AI)
- [ ] 영감이 실제로 도움이 되는가?
- [ ] 너무 추상적이지 않은가?
- [ ] 실행 가능한 형태인가?
- [ ] 대상자의 상황에 맞는가?

### Claude Code (Development)
- [ ] 코드가 에러 없이 실행되는가?
- [ ] 테스트를 통과했는가?
- [ ] 보안 취약점이 없는가?
- [ ] 기존 코드와 스타일이 일관적인가?
- [ ] 불필요한 코드가 없는가?

## 배드 시그널 목록

### 서비스 관련

| 시그널 | 심각도 | 대응 |
|--------|--------|------|
| 소원이가 결과 보고 바로 이탈 | 🔴 Critical | 즉시 UX 검토 |
| 응원 메시지가 기계적으로 느껴짐 | 🟡 Warning | 프롬프트 개선 |
| 유료 전환율 5% 미만 | 🟡 Warning | 가치 제안 재검토 |
| NPS 50 미만 | 🔴 Critical | 전체 서비스 점검 |
| 재방문율 20% 미만 | 🟡 Warning | 리텐션 전략 수립 |

### 운영 관련

| 시그널 | 심각도 | 대응 |
|--------|--------|------|
| 일일 보고서를 안 읽게 됨 | 🟡 Warning | 포맷/내용 개선 |
| 에이전트 결과물을 일일이 수정해야 함 | 🔴 Critical | 지침 대폭 보강 |
| 같은 오류가 3번 이상 반복 | 🔴 Critical | 근본 원인 해결 |
| 수동 개입이 하루 5회 이상 | 🟡 Warning | 자동화 확대 |

### 기술 관련

| 시그널 | 심각도 | 대응 |
|--------|--------|------|
| API 에러율 5% 이상 | 🔴 Critical | 즉시 디버깅 |
| 응답 시간 3초 이상 | 🟡 Warning | 최적화 |
| 메모리 누수 징후 | 🔴 Critical | 코드 리뷰 |

## 릴리즈 전 체크리스트

### 필수 항목 (모두 통과해야 배포)

- [ ] 도그푸딩 완료 (푸르미르님 직접 사용)
- [ ] 에이전트 검증 로직 포함
- [ ] 피드백 반영 완료
- [ ] 최종 종합 검증
- [ ] 배드 시그널 제로 (또는 known issue 문서화)

### 권장 항목

- [ ] 부하 테스트 완료
- [ ] 롤백 계획 수립
- [ ] 모니터링 대시보드 준비
- [ ] 긴급 연락 체계 확인

## 자가 검증 템플릿

에이전트 .md 파일에 추가할 섹션:

```markdown
## 자가 검증

작업 완료 후 다음을 스스로 확인하라:

1. [ ] [체크항목 1]
2. [ ] [체크항목 2]
3. [ ] [체크항목 3]

### 검증 실패 시
- 문제점 기록
- 재작업 실행
- 다시 검증
- 3회 실패 시 에스컬레이션
```

## 지속적 개선 사이클

```
검증 → 발견 → 기록 → 개선 → 검증 (반복)
         ↓
   피드백 기록에 추가
         ↓
   지침에 영구 반영
```

**핵심**: 같은 실수를 두 번 하지 않는다.
